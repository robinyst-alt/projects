## 一、项目背景

### 1.1 市场痛点

- 中国老龄化加速，独居老人超1.18亿
- 老人跌倒后无人知，错过黄金救援时间（30分钟内）
- 亲属工作繁忙，无法实时陪伴
- 现有养老设备操作复杂，老人不会用

### 1.2 解决方案

用AI语音交互降低使用门槛，通过摄像头实时检测跌倒，实现"跌倒感知 → AI问询 → 无响应则通知亲属"的自动化救援流程。

### 1.3 产品定位

| 维度 | 说明 |
|------|------|
| 产品名称 | 养老AI助手 |
| 核心定位 | 独居老人的紧急救援 + 日常陪伴 |
| MVP定位 | 电脑上演示版，验证核心场景，未来移植树莓派 |

## 二、商业模式与角色

### 2.1 核心角色

| 角色 | 说明 | 付费方式 |
|------|------|---------|
| **老人（主要用户）** | 使用设备，进行语音对话 | 硬件销售 / 月租 |
| **亲属（次要用户）** | 接收告警、查看状态、远程配置 | 免费（增值服务收费） |

| **合作伙伴（未来）** | 120中心、医院、养老院 | API调用分成 |

### 2.2 价值流

```
服务商 ──提供设备和云服务──▶ 老人
                                      │
亲属 ◀──接收告警+远程管理──┘
         │
         └──反向联系老人（TTS播报）
```

### 4.2 AI意图识别与服务扩展

> ⚠️ 以下为服务商端（未来）功能规划，MVP阶段不实现。

AI不仅响应用户对话，还需**理解用户意图**并调用对应服务：

| 用户需求 | AI处理方式 | 服务来源 |
|---------|-----------|---------|
| "今天天气怎么样" | AI调用天气API | wttr.in / 气象局API |
| "提醒我明天9点开会" | AI创建日程提醒 | 日历服务 |
| "帮我叫个外卖" | AI联系外卖服务 | 合作伙伴API |
| "我想听新闻" | AI播报新闻 | 新闻API |
| "打给闺女" | 识别意图+多联络人确认+拨打电话 | 通讯录 |

**服务商端开发要点**：
- 分析主要用户交互需求，定向开发确保回复准确性
- 每个服务意图需单独训练/调试，确保持久稳定性
- 服务商端负责维护服务API的质量和可用性

### 4.3 数据流

#### 跌倒检测数据流

```
摄像头 → MediaPipe Pose → 跌倒判定
    │                           │
    │                    触发关怀流程
    │                           │
    │                    AI问询（3次×20秒）
    │                           │
    │              ┌────────────┴────────────┐
    │              ▼                         ▼
    │        用户响应OK              无响应→通知亲属
    │              │                         │
    │              ▼                         ▼
    │        记录事件+继续监控         Webhook POST亲属
    │                                        │
    │（同步）                             （异步）
    ▼                                        ▼
事件日志                                 亲属APP收到推送
```

#### 亲属通知数据流

```
老人设备 ──POST──▶ 亲属APP（Webhook URL）
                        │
                        ├── 消息类型：fall_detected / sos / medication_reminder
                        ├── 老人信息：姓名、位置
                        ├── 事件详情：时间、置信度、轨迹
                        └── 亲属操作：确认已处理 / 回拨 / 联系120
```

#### 事件长期存储

事件历史在设备端保留5年，超过5年的事件迁移至云端服务器存储，设备端不再显示。详见【老人端PRD 2.5.2事件历史】。

## 五、统一API契约

### 5.1 老人端暴露的API

| 方法 | 路径 | 说明 | 调用方 |
|------|------|------|--------|
| POST | `/chat` | AI对话 | 老人端UI |
| POST | `/stt` | 语音转文字 | 老人端UI |
| POST | `/tts` | 文字转语音 | 老人端UI |
| POST | `/fall/start` | 开启跌倒检测 | 老人端UI |
| POST | `/fall/stop` | 停止跌倒检测 | 老人端UI |
| GET | `/pose/trajectory` | 获取运动轨迹 | 老人端UI |
| POST | `/fall/response` | 跌倒后用户响应 | 老人端UI |
| POST | `/fall/cancel` | 取消紧急联系 | 老人端UI |
| GET | `/fall/escalation_status` | 关怀流程状态 | 老人端UI |
| GET | `/events` | 查看事件日志 | 老人端UI / 亲属端 |
| GET/POST | `/user/profile` | 用户资料 | 老人端UI / 亲属端 |
| GET/POST | `/ai/profile` | AI称呼 | 老人端UI |
| GET/POST | `/medication/reminders` | 用药提醒 | 老人端UI / 亲属端 |
| POST | `/sos` | 触发紧急呼救 | 老人端UI |
| GET/POST | `/notification/settings` | 通知设置 | 亲属端 |

### 5.2 亲属端Webhook回调格式

老人设备主动 POST 到亲属端：

```json
POST {亲属端注册的Webhook URL}
Content-Type: application/json

{
  "event_type": "fall_detected",
  "device_id": "device-001",
  "elder_name": "王大爷",
  "location": "客厅",
  "timestamp": "2026-04-25T10:30:00Z",
  "fall_confidence": 0.85,
  "trajectory": [
    {"timestamp": 1745337580.1, "pose": "standing"},
    {"timestamp": 1745337581.2, "pose": "falling"}
  ],
  "ai_response": "检测到跌倒，已联系您的亲属"
}
```

支持的事件类型：`fall_detected` / `sos_triggered` / `medication_reminder` / `care_reminder`

## 六、技术选型（总体）

| 组件 | 选型 | 说明 |
|------|------|------|
| LLM | Ollama + gemma4:e4b | 本地运行，响应快，隐私安全 |
| STT | Whisper (openai-whisper) | 中文语音识别 |
| TTS | espeak | 本地语音合成，无需网络 |
| 跌倒检测 | MediaPipe Pose | 33个关节点，边缘可运行 |
| 老人端UI | Tkinter (MVP) / Web (未来) | 演示用Tkinter，未来换Web |
| 亲属端APP | 微信小程序 / React Native | 待定 |
| 定时任务 | APScheduler | 轻量级，支持持久化 |
| 摄像头 | OpenCV | 成熟稳定，跨平台 |
| 配置存储 | JSON文件 | 轻量级，无需数据库 |
| 云端服务 | FastAPI + 云数据库 | 服务商部署 |
