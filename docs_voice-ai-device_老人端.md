## 一、用户画像

- **主要用户**：独居老人（65岁以上），子女不在身边
- **使用场景**：日常对话、跌倒检测紧急救援、用药提醒
- **操作能力**：可能不熟悉手机/电脑操作，语音交互为主
- **痛点**：孤独感、跌倒无人知、忘记吃药

## 二、UI/UX设计

### 2.1 设计原则

- **语音为主，按钮为辅**：所有功能（对话、联系子女、紧急呼救）均可通过纯语音交互实现；按钮是给习惯操作按钮的老人使用的补充方式
- **纯语音交互**：用户不通过键盘/输入法与AI对话，所有交互通过语音完成
- **屏幕辅助查看**：配置结果、事件记录、AI回复等通过屏幕展示给用户
- **最低操作门槛**：无需打字，老人只需说话即可完成所有操作
- **AI询问简洁**：AI询问用户时不要啰嗦，展示「xxx吗？」或「xxx请确认」即可，后台默认关键词逻辑（如"好的"确认、"不是"修改）
- **双向同步输出**：用户说的话实时显示文字在聊天区；AI回复同时输出文字+语音

### 2.2 页面结构


### 2.3 交互模式

#### 2.3.1 唤醒

用户通过唤醒词（连续呼唤AI名字两遍）唤醒AI，AI回复"我在"。

> 唤醒功能详细说明，见第三章「语音交互」。

#### 2.3.2 日常对话

唤醒后，用户直接说话与AI对话。按钮为辅助方式。

> 未来扩展：AI接入天气查询等服务，调用外部API。

#### 2.3.3 联系子女

- **触发**：用户说"联系子女"、"打电话给儿子"、"打给闺女"等类似表达
- **意图识别**：AI通过LLM理解用户意图，不依赖固定词
- **默认联络逻辑**：除非用户特指打给谁，否则按以下优先级联络：
  1. 优先联络人（设置中指定）
  2. 优先通信方式（微信电话/普通电话）
- **多联络人确认**：如果紧急联络人≥2个且用户未指定，AI确认"您想联系哪位？"
  - 例："大儿子"、"二女儿"、"张三"
- **联络复述**：AI联系时同步复述，如"好的，我现在使用微信电话打给大儿子，请稍后"
- **按钮方式**：点击蓝色「联系子女」按钮 → 弹出【微信电话】【普通电话】两个选项 → 用户选择后拨打

#### 2.3.4 创建用药提醒

- **触发**：用户说"提醒我每天8点吃降压药"
- **意图识别**：AI解析时间、药品名称
- **Confirm确认**："好的，我帮您设置每天早上8点提醒您吃降压药，请确认"
- **Confirm通过后**：写入定时任务（详见2.6设置页「用药提醒」）

#### 2.3.5 修改个人信息

- **触发**：用户说"我叫张三"、"我68岁"等
- **意图识别**：AI识别需要修改的字段和新值（姓名、年龄、病史、电话、微信、地址、紧急联络人等，见2.6设置页详细说明）
- **Confirm确认**："好的，我帮您把姓名改成张三，请确认"
- **Confirm通过后**：更新用户资料

#### 2.3.6 紧急呼救

紧急呼救与其他模式不同，**本身即唤醒词**，无需先唤醒AI。

- **触发词**："救命"、"打120"、"紧急呼救"、"我摔倒了"等
- **无需唤醒**：直接进入紧急呼救流程（详见【5.3主动触发：紧急呼救流程】）

#### 2.3.7 查看事件/配置

- **查看内容**：事件历史（详见【2.5事件历史】）、用药提醒列表（详见【2.6设置页】）、个人信息（详见【2.6设置页】）

## 三、语音交互

### 3.1 语音唤醒 + 对话流程

```
日常待机状态
    ↑
    │ 用户连续呼唤AI名字两遍（如"小管家小管家"）
    │
    ↓ 唤醒成功
AI立即回复"我在"
    ↓
进入对话状态
    ↓
用户说话 → Whisper STT → LLM处理 → AI回复
    ↓
对话结束（如30秒无活动）→ 返回待机状态
```

### 3.2 唤醒词检测

| 参数 | 值 | 说明 |
|------|-----|------|
| 唤醒词 | AI名字 × 2 | 如"小管家小管家" |
| 检测方式 | Whisper STT | 实时监听并检测唤醒词 |
| 响应延迟 | < 500ms | 唤醒后立即回复"我在" |
| 待机模式 | 屏幕熄灭，AI不主动响应 | 降低功耗；唤醒词、触屏、按钮操作均可点亮屏幕 |

### 3.3 AI自动填写

对话中AI自动识别意图并操作：

| 用户说 | AI自动 |
|--------|--------|
| "提醒我每天8点吃降压药" | 创建用药提醒 |
| "我叫张三" | 更新姓名 |
| "我68岁" | 更新年龄 |
| "提醒我下午2点吃消炎药" | 创建14:00用药提醒 |

## 四、跌倒检测算法

### 4.1 检测逻辑

使用 MediaPipe Pose 分析躯干角度和头部高度：

```python
# 关键阈值
FALL_ANGLE_THRESHOLD = 45.0  # 躯干与垂直方向夹角 > 45° 视为跌倒
HEIGHT_DROP_RATIO = 0.35     # 高度下降 > 35% 视为跌倒
CONFIDENCE_THRESHOLD = 0.6    # 姿态检测置信度 > 0.6 才采纳
```

### 4.2 跌倒判定条件

同时满足以下条件判定为跌倒：
1. 躯干角度 > 45°（身体倾斜）
2. 头部高度下降 > 35%（相对于站立基准）
3. 持续时间 > 0.5秒（排除短暂弯腰）

### 4.3 姿态分类

| 条件 | 姿态 |
|------|------|
| 躯干角度 > 60° | 【躺着】 |
| 躯干角度 > 30° 且膝盖与臀部高度接近 | 【坐着】 |
| 躯干角度 > 30° 且不符合坐着条件 | 【跌倒】 |
| 脚踝高度周期性变化 > 0.08 | 【行走】 |
| 脚踝低于臀部 > 0.15 | 【站立】 |

## 五、呼救流程

### 5.1 框架概述

呼救流程按**触发类型**分为两类：

| 触发类型 | 说明 | 详见 |
|---------|------|------|
| **被动触发** | MediaPipe检测到跌倒姿态 | 【4.跌倒检测算法】 |
| **主动触发** | 用户喊出紧急呼救触发词 | 【5.3主动触发：紧急呼救流程】 |

按**Action类型**分为两种：

| Action类型 | 说明 | 详见 |
|-----------|------|------|
| **联系子女** | Webhook通知子女 | 【5.4联系子女流程】 |
| **拨打120** | 直接通知急救中心 | 【5.5拨打120流程】 |

AI询问用户时给出选择：「是要我联系您的子女，还是直接拨打120？」

### 5.2 被动触发：跌倒检测流程

跌倒判定基于MediaPipe Pose（详见【4.跌倒检测算法】）。

```
检测到跌倒（MediaPipe Pose 确认）
    ↓
记录事件 + 轨迹（最近30秒姿态数据，详见【6.运动轨迹记录】）
    ↓
立即触发语音唤醒
"我在"（≤2秒）
    ↓
"您没事吧？"（≤2秒）
    ↓ （等待20秒）
无响应 → "您摔倒了吗？是要我联系您的子女，还是直接拨打120？"
    ↓ （等待20秒）
无响应 → "我准备联系您的子女了。"
    ↓ （等待20秒）
无响应 → 联系子女 + 记录事件
```

**用户响应处理**：

| 响应 | 含义 | 处理 |
|------|------|------|
| 没事 | 确认无恙 | 取消流程，继续监测 |
| 联系子女 | 选择Action | 联系子女 |
| 打120 | 选择Action | 拨打120 |
| cancel | 取消 | 取消流程 |

### 5.3 主动触发：紧急呼救流程

紧急呼救**本身即唤醒词**，无需先唤醒AI。

**触发词**："救命"、"打120"、"紧急呼救"、"我摔倒了"、"快来人"等。

```
用户喊出触发词
    ↓
AI立即回复（≤1秒）
"我在，您哪里不舒服？"
    ↓ （等待20秒）
无响应 → "您是需要帮助吗？是要我联系您的子女，还是直接拨打120？"
    ↓ （等待20秒）
无响应 → "我准备联系您的子女了。"
    ↓ （等待20秒）
无响应 → 联系子女 + 记录事件
```

**用户响应处理**：

| 响应 | 含义 | 处理 |
|------|------|------|
| 是的/确认/好 | 确认紧急 | 执行用户选择的Action |
| 没事/不用了 | 撤销 | 取消流程 |
| cancel | 取消 | 取消流程 |
| 联系子女 | 选择Action | 联系子女 |
| 打120 | 选择Action | 拨打120 |

### 5.4 联系子女流程

```
确定联络对象（用户指定 / 优先联络人）
    ↓
确定通信方式（用户指定 / 优先联络方式：微信电话/普通电话）
    ↓
AI复述
"好的，我使用{微信电话/电话}打给{优先联络人姓名}，请稍后。"
    ↓
执行联络（Webhook通知子女）
    ↓
记录事件【联系子女：xxx】（详见【2.5事件历史】）
```

### 5.5 拨打120流程

```
AI复述
"好的，我现在拨打120，请稍后。"
    ↓
触发SOS（Webhook通知子女 + 拨打120）
    ↓
记录事件【紧急呼救】（详见【2.5事件历史】）
```

### 5.6 关键时序要求

| 阶段 | 时限 |
|------|------|
| 跌倒确认 → "我在" | ≤ 2秒 |
| "我在" → 第1次询问 | ≤ 2秒 |
| 每次询问等待 | 20秒 |
| 无响应 → 触发Action | 60秒 |

### 5.7 流程状态

| 状态 | 说明 |
|------|------|
| IDLE | 无活动 |
| CHECKING_IN | 正在询问（1-3次） |
| ACTIONING | 正在执行联络 |
| RESOLVED | 已解决 |

## 六、运动轨迹记录

### 6.1 数据结构

```python
# 滚动缓冲区：最近5分钟（300条）
MAX_TRAJECTORY_BUFFER = 300

{
    "timestamp": 1745337580.123,  # Unix时间戳
    "pose": "standing"            # 姿态状态
}
```

### 6.2 轨迹用途

- 跌倒事件回溯：跌倒发生前后各30秒的姿态序列
- 事件日志存储：跌倒事件包含 `trajectory` 字段
- 查询API：`GET /pose/trajectory?seconds=30`

## 九、验收标准

| 测试项 | 通过标准 |
|--------|---------|
| 语音唤醒 | 连续呼唤AI名字两遍，2秒内回复"我在" |
| AI回复同步 | 文字+语音同时输出，用户能看到也能听到 |
| 语音对话 | 唤醒后说话，AI响应<3s，同步文字+语音 |
| 跌倒检测 | 模拟跌倒动作，5次中至少4次检测到 |
| 关怀时序 | 跌倒确认到"我在"≤2秒，"我在"到第1次询问≤2秒 |
| 关怀流程 | 检测到跌倒后，AI询问3次×20秒，无响应触发SOS |
| 轨迹记录 | 跌倒事件包含最近30秒姿态数据 |
| AI自动创建提醒 | 说"提醒我每天8点吃降压药"，定时触发 |
| 用户响应 | 说"没事"则取消SOS，说"需要帮助"则立即联系子女 |
