## 三、UI/UX设计

### 3.1 设计原则

- **语音为主，按钮为辅**：所有功能（对话、联系亲属、紧急呼救）均可通过纯语音交互实现；按钮是给习惯操作按钮的老人使用的补充方式
- **纯语音交互**：用户不通过键盘/输入法与AI对话，所有交互通过语音完成
- **屏幕辅助查看**：配置结果、事件记录、AI回复等通过屏幕展示给用户
- **最低操作门槛**：无需打字，老人只需说话即可完成所有操作
- **AI询问简洁**：AI询问用户时不要啰嗦，展示「xxx吗？」或「xxx请确认」即可，后台默认关键词逻辑（如"好的"确认、"不是"修改）
- **准同步输出**：用户说的话实时显示文字在聊天区；AI回复时，文字先通过LLM流式输出显示，TTS后台合成后播放语音（文字略先于语音，用户感知为同步）

### 3.2 页面结构

```
养老AI助手（Tkinter）
├── 对话页（默认）
│   ├── AI名称 + 连接状态
│   ├── 聊天记录区（AI回复 + 系统消息）
│   ├── 【按住说话】【联系亲属】【紧急呼救】按钮
│   │   └── ⚠️ 按钮为辅助，核心交互通过语音（唤醒词）完成
│   └── ⚠️ 无文字输入框：用户不打字
│
├── 事件页
│   ├── 事件历史（格式：【时间】【事件】）
│   └── 刷新按钮
│
├── 用药页
│   ├── 添加用药提醒（语音创建：告诉AI即可）
│   └── 提醒列表（可查看配置结果）
│
└── 设置页（分栏显示）
    ├── 第一栏：AI助手（称呼）
    ├── 第二栏：老人信息（姓名、年龄、病史、电话、微信、地址）
    ├── 第三栏：紧急联络人（姓名+电话+微信，≤3人，含优先联络人选项）
    └── 第四栏：联络设置（优先微信电话联络）
```

### 3.3 交互模式

#### 3.3.1 唤醒

用户通过唤醒词（连续呼唤AI名字两遍）唤醒AI，AI回复"我在"。

> 唤醒功能详细说明，见【../03-语音交互.md-4.1】。

#### 3.3.2 日常对话

唤醒后，用户直接说话与AI对话。按钮为辅助方式。

> 未来扩展：AI接入天气查询等服务，调用外部API。

#### 3.3.3 联系亲属

- **触发**：用户说"联系亲属"、"打电话给儿子"、"打给闺女"等类似表达
- **意图识别**：AI通过LLM理解用户意图，不依赖固定词
- **默认联络逻辑**：除非用户特指打给谁，否则按以下优先级联络：
  1. 优先联络人（设置中指定）
  2. 优先通信方式（微信电话/普通电话）
- **多联络人确认**：如果紧急联络人≥2个且用户未指定，AI确认"您想联系哪位？"
  - 例："大儿子"、"二女儿"、"张三"
- **联络复述**：AI联系时同步复述，如"好的，我现在使用微信电话打给大儿子，请稍后"
- **按钮方式**：点击蓝色「联系亲属」按钮 → 弹出【微信电话】【普通电话】两个选项 → 用户选择后拨打

#### 3.3.4 创建用药提醒

- **触发**：用户说"提醒我每天8点吃降压药"
- **意图识别**：AI解析时间、药品名称
- **Confirm确认**："好的，我帮您设置每天早上8点提醒您吃降压药，请确认"
- **Confirm通过后**：写入定时任务（详见【3.6设置页】）

#### 3.3.5 修改个人信息

- **触发**：用户说"我叫张三"、"我68岁"等
- **意图识别**：AI识别需要修改的字段和新值（姓名、年龄、病史、电话、微信、地址、紧急联络人等，见【3.6设置页详细说明】）
- **Confirm确认**："好的，我帮您把姓名改成张三，请确认"
- **Confirm通过后**：更新用户资料

#### 3.3.6 紧急呼救

紧急呼救与其他模式不同，**本身即唤醒词**，无需先唤醒AI。

- **触发词**："救命"、"打120"、"紧急呼救"、"我摔倒了"等
- **无需唤醒**：直接进入紧急呼救流程（详见【../04-感知与检测.md-6.3】）

#### 3.3.7 查看事件/配置

- **查看内容**：事件历史（详见【3.5事件历史】）、用药提醒列表（详见【3.6设置页】）、个人信息（详见【3.6设置页】）

### 3.4 关键UI元素

| 元素 | 样式 | 说明 |

| 按住说话按钮 | 粉红色 #f38ba8 | 辅助：语音输入 |

### 3.5 事件历史

#### 3.5.1 记录格式

**【时间】【事件】**，如：`【10:30】【用药提醒：降压药】`

#### 3.5.2 记录内容（UI显示）

事件历史采用**多级展开模式**，默认展示最近3天：

```
▼ 2026-04-26（今天）[默认展开]
  【10:30】【用药提醒：降压药】
  【10:25】【跌倒检测】
  【10:20】【联系亲属：大儿子】
  【09:00】【紧急呼救】

▶ 2026-04-25
▶ 2026-04-24
▶ 最近7天
▶ 最近1个月
▶ 最近6个月
▶ 最近1年
▶ 最近5年
```

**展开层级规则**：

| 时间范围 | 展开层级 | 操作 |
|---------|---------|------|
| 最近3天 | 按天展开（最细粒度） | 默认展开最近3天，点击可收起/展开 |
| 最近7天 | 按天展开 | 点击展开后显示逐天 |
| 最近1个月 | 按天展开 | 超过7天后按月显示，逐月展开 |
| 最近6个月 | 按月展开 | 点击展开月，显示该月内逐日 |
| 最近1年 | 按月展开 | 同上 |
| 最近5年 | 按年展开 | 点击展开年，显示该年内逐月 |
| 5年以上 | 不显示 | 存储在服务端（详见【../06-技术实现.md-9.2】） |

**交互说明**：
- ▶ 表示折叠态，点击展开
- ▼ 表示展开态，点击折叠
- 默认只展开最近3天
- 存储上限5年，超过存储在云端

| 事件类型 | 格式示例 | 说明 |
|---------|---------|--------|
| 用药提醒 | `【10:30】【用药提醒：降压药】` | 定时触发 |
| 跌倒检测 | `【10:25】【跌倒检测】` | 检测到跌倒 |
| 亲属通话 | `【10:20】【联系亲属：大儿子】` | 用户触发联系亲属 |
| 紧急呼救 | `【10:15】【紧急呼救】` | 用户触发SOS |

#### 3.5.3 不记录（后台存储）

以下数据**不显示在UI事件历史**中，但**后台持续记录**，供亲属端监控查看：
- 日常语音对话内容
- AI回复内容
- 老人活动状态（由亲属端主动查询）

| 姿态 | 显示 |
|------|------|
| standing | 🧍【站立】 |
| sitting | 🧍【坐下】 |
| walking | 🚶【行走】 |
| lying_down | 😴【躺着】 |
| falling | 🤕【跌倒】 |
| unknown | ❓【未知】 |

### 3.6 设置页详细说明

#### 第一栏：AI助手

| 字段 | 类型 | 说明 |
|------|------|------|
| AI称呼 | 文本显示 | 如"小管家" |

#### 第二栏：老人信息

| 字段 | 类型 | 说明 |
|------|------|------|
| 姓名 | 文本 | 老人的名字 |
| 年龄 | 数字 | 岁数 |
| 病史 | 文本 | 健康状况、既往病史 |
| 电话 | 文本 | 老人联系电话 |
| 微信 | 文本 | 老人微信（用于亲属微信联络） |
| 地址 | 文本 | 家庭住址 |

#### 第三栏：紧急联络人

| 字段 | 类型 | 说明 |
|------|------|------|
| 紧急联络人 | 动态列表 | 姓名 + 关系 + 电话 + 微信，最少1人，最多3人 |
| 优先联络人 | 复选框 | 每行一个，勾选后其他联络人的此选项灰掉（只能选1人） |
| 增加按钮 | 按钮 | 点击新增一个紧急联络人 |
| 删除按钮 | 按钮 | 每个联络人可删除 |

**优先联络人逻辑**：
- 勾选某联络人为优先后，其他联络人的「优先联络人」选项自动灰掉
- 「联系亲属」按钮按此人设置执行

#### 第四栏：联络设置（全局）

| 字段 | 类型 | 说明 |
|------|------|------|
| 优先微信电话联络 | 复选框 | 打勾 → 优先使用微信电话；不打勾 → 优先使用普通电话 |

**增加限制逻辑**：
- 当联络人数量 ≥ 3 时，"增加"按钮**禁用**
- 当用户尝试增加第4个联络人时，页面**红字提示**：`【紧急联络人数量已超过上限（3人）】`

#### 第五栏：设备关联

| 字段 | 类型 | 说明 |
|------|------|------|
| 关联手机 | 文本显示 | 显示已关联的老人手机名称/型号 |
| 连接状态 | 状态指示 | 已连接/未连接/连接中 |
| 配对按钮 | 按钮 | 点击进入蓝牙配对流程 |
| 测试拨打 | 按钮 | 测试电话拨打功能是否正常 |

**电话拨打说明**：
- 通过蓝牙连接老人手机，从设备发起普通电话呼叫
- 支持拨打120急救电话、亲属电话
- 触发条件：紧急呼救流程、联系亲属（选择普通电话时）
- 需要老人授权手机通话权限
