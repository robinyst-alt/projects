# 子女APP - 产品规划

> 整理时间：2026-04-24
> 状态：规划阶段

---

## 一、产品定位

| 维度 | 说明 |
|------|------|
| 产品名称 | 养老AI助手（子女端） |
| 核心定位 | 子女接收老人设备推送的消息、查看老人状态、远程配置设备 |
| 与老人设备关系 | 老人设备的消息接收方和远程管理端 |

---

## 二、核心功能

### 2.1 消息接收（P0 - 必须有）

子女APP本质是一个**消息推送接收端**，主要接收来自老人设备的消息：

| 消息类型 | 说明 | 推送时机 |
|---------|------|---------|
| 跌倒告警 | 设备检测到跌倒，AI询问无响应后触发 | 自动 |
| SOS紧急呼救 | 老人语音触发"打120" | 手动/自动 |
| 子女联系请求 | 设备按下"联系子女"按钮 | 手动 |
| 每日健康报告 | 汇总一天的活动轨迹、异常事件 | 每日定时 |

推送方式：
- **基础方案**：轮询设备Webhook接口（老人设备作为服务端）
- **进阶方案**：通过消息队列/推送服务（如极光推送、Firebase Cloud Messaging）

### 2.2 事件查看（P0 - 必须有）

子女可以查看老人的事件历史：

| 功能 | 说明 |
|------|------|
| 事件列表 | 按时间倒序显示所有事件（跌倒、SOS、姿态变化、用药提醒等） |
| 事件详情 | 点击查看单条事件的完整信息（包含跌倒轨迹数据） |
| 事件筛选 | 按类型筛选（仅跌倒、仅SOS、仅用药提醒等） |
| 事件时间线 | 可视化展示老人一天的活动状态 |

### 2.3 设备配置（P1 - 应该有）

子女可以远程配置老人设备：

| 配置项 | 说明 |
|-------|------|
| 紧急联系人设置 | 姓名、电话（最多3个） |
| 设备信息 | 老人姓名、房间位置 |
| 提醒阈值 | 跌倒检测灵敏度、询问等待时间 |
| 用药提醒管理 | 查看/新增/删除老人的用药提醒 |
| AI称呼设置 | 修改AI助手的名字 |

### 2.4 双向消息（P1 - 应该有）

| 功能 | 说明 |
|------|------|
| 给老人发消息 | 子女发送文字/语音给老人设备，设备TTS播报 |
| 语音留言 | 子女录制语音推送到设备 |

## 三、技术方案

### 3.1 架构选择

**方案A：React Native / Flutter 跨平台APP**（推荐）
- 一次开发，iOS/Android 都能用
- 开发成本适中
- 可以调用设备原生能力（推送通知、麦克风等）

**方案B：微信小程序**
- 用户无需安装，扫码即用
- 推送依赖微信订阅消息，限制多
- 适合快速验证

**方案C：Web APP（PWA）**
- 直接复用老人设备的Web API
- 可以添加到手机桌面
- 推送能力弱

### 3.2 数据流

```
老人设备（FastAPI）                    子女APP
      │                                  │
      │  ──HTTP POST Webhook────────>    │  接收推送
      │                                  │
      │  <──轮询 /events───────────       │  拉取事件
      │                                  │
      │  <──PUT /settings───────────      │  远程配置
      │                                  │
      │  <──POST /message──────────      │  发送消息给老人
```

### 3.3 关键API（老人设备端）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/notification/webhook` | 消息推送（子女端提供URL，设备回调） |
| GET | `/events` | 获取事件列表 |
| GET | `/events/{id}` | 获取事件详情（含轨迹） |
| GET | `/user/profile` | 获取老人信息 |
| PUT | `/user/profile` | 更新老人信息 |
| GET | `/medication/reminders` | 获取用药提醒 |
| POST | `/medication/reminders` | 添加用药提醒 |
| DELETE | `/medication/reminders/{id}` | 删除用药提醒 |
| GET | `/care/settings` | 获取关怀设置 |
| PUT | `/care/settings` | 更新关怀设置 |
| POST | `/device/message` | 给老人发送消息（文字/语音） |

## 四、界面设计

### 4.1 页面结构

```
子女APP
├── 首页/消息列表（默认）
│   ├── 紧急告警横幅（跌倒/SOS时置顶）
│   └── 事件卡片列表
│
├── 事件详情
│   ├── 事件基本信息（类型、时间、老人信息）
│   ├── 跌倒轨迹（如果有）
│   └── 操作按钮（确认已处理/回拨/联系120）
│
├── 设备配置
│   ├── 老人信息（姓名、位置）
│   ├── 紧急联系人
│   ├── 跌倒检测灵敏度
│   ├── 用药提醒列表（增删改）
│   └── AI称呼
│
└── 我的
    ├── 账号信息
    ├── 消息通知设置
    └── 关于/帮助
```

### 4.2 视觉风格

- 简洁、温暖的设计语言
- 紧急事件用红色/橙色醒目提示
- 事件卡片展示关键信息，详情页展示完整数据
- 考虑中老年子女的字体大小（至少14sp）

## 五、优先级规划

| 阶段 | 功能 | 说明 |
|------|------|------|
| **MVP** | 消息接收 + 事件列表 + 事件详情 | 子女能看到老人的所有事件 |
| **V1.1** | 设备配置（老人信息、紧急联系人） | 子女可远程管理设备 |
| **V1.2** | 用药提醒管理 | 子女帮老人设置用药提醒 |
| **V2.0** | 双向消息 + 每日健康报告 | 增强互动性和健康监测能力 |

## 六、待讨论问题

1. **推送方式**：设备作为服务端主动推送 vs 子女APP轮询？
   - 设备在内网时需要穿透，建议用轮询+Webhook混合

2. **账号体系**：子女是否需要登录？
   - 建议：子女扫码绑定设备，无需注册账号

3. **多子女支持**：一个老人，多个子女都能收到消息吗？
   - 需要：Webhook支持多个URL，或者消息队列广播

4. **数据存储**：事件历史存储在哪里？
   - 建议：存储在老人设备本地，子女APP按需拉取
   - 长期历史可以考虑云端备份
